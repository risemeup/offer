## 核心薄弱点一：InnoDB 锁机制深度解析

### 1. 三种行级锁的区别
*   **记录锁 (Record Lock)**：仅锁住索引记录本身。用于唯一索引/主键的精确等值匹配且记录存在时。
*   **间隙锁 (Gap Lock)**：锁住两条记录之间的**开区间 `(a, b)`**。纯粹为了防止其他事务在这个间隙里 `INSERT` 新数据（防幻读第一道防线）。间隙锁之间不互斥。
*   **临键锁 (Next-Key Lock)**：**记录锁 + 间隙锁**。锁住记录本身以及前面的间隙，是一个**左开右闭区间 `(a, b]`**。这是 RR 隔离级别下加锁的**默认基本单位**。

### 2. 临键锁的“退化”规则（必考重点）
在 RR 隔离级别下，执行等值查询（`=`）时：
1.  **唯一索引 + 数据存在**：退化为 **记录锁**（只锁精确的一行）。
2.  **唯一索引 + 数据不存在**：退化为 **间隙锁**（只锁目标值所在的空隙）。
3.  **非唯一索引 + 数据存在**：**不退化**，保持 **临键锁 + 间隙锁**（锁前面的间隙 + 记录本身 + 向右遍历到下一个不相等的值并加间隙锁）。
4.  **非唯一索引 + 数据不存在**：退化为 **间隙锁**（只锁目标值所在的空隙）。

> **💡 灵魂拷问：为什么非唯一索引数据存在时，还要锁右边的间隙？**
> 因为非唯一索引允许重复。当索引值相同时，B+树会根据主键ID排序。如果不锁右边的间隙，其他事务可能会插入一条 `索引值相同，但主键ID更大` 的新数据（这会被排在当前记录的右侧），从而导致幻读。因此必须前后包夹。

### 3. 行锁升级为表锁的场景
*   **原因**：InnoDB 的行锁是加在**索引项**上的，而不是数据行本身。
*   **场景**：如果 `UPDATE/DELETE` 等操作的 `WHERE` 条件**没有命中索引**（或索引失效），MySQL 无法精确定位索引节点，只能进行**全表扫描**。
*   **后果**：在全表扫描过程中，会给扫描过的所有行的聚簇索引加锁，宏观效果上等同于**锁死了整张表**，严重拖垮并发性能。

## 核心薄弱点二：日志与两阶段提交

### 1. 三大核心日志
*   **Undo Log（回滚日志）**：引擎层生成，记录修改**前**的数据。用于**事务回滚（保证原子性）**和**实现 MVCC 版本链**。
*   **Redo Log（重做日志）**：引擎层生成，记录修改**后**的物理变化。用于**宕机恢复（Crash-safe，保证持久性）**。
*   **Binlog（归档日志）**：Server 层生成，记录逻辑 SQL 变更。用于**主从复制**和**数据备份恢复**。

### 2. 两阶段提交 (Two-Phase Commit)
*   **目的**：保证 Redo Log 和 Binlog 的数据一致性。
*   **流程**：
    1.  **Prepare 阶段**：写入 Redo Log，状态标记为 Prepare。
    2.  **Commit 阶段**：写入 Binlog，随后将 Redo Log 状态改为 Commit。
*   **如果不这么做**：无论先写谁，只要写完第一个就宕机，都会导致主库（靠 Redo Log 恢复）和从库（靠 Binlog 同步）的数据不一致。

## 核心薄弱点三：死锁与并发实战

### 1. 死锁 (Deadlock)
*   **定义**：两个或多个事务互相持有对方需要的锁并等待对方释放，导致无限期阻塞。
*   **经典场景（交叉更新）**：事务A锁id=1等id=2，事务B锁id=2等id=1。
*   **MySQL 处理机制**：默认开启死锁检测（`innodb_deadlock_detect=on`），发现死锁回路后，主动**回滚成本较小**（改动行数少）的事务。

### 2. 秒杀防超卖经典 SQL
```sql
UPDATE stock SET count = count - 1 WHERE id = 1 AND count > 0;
```
*   **原理**：利用 `UPDATE` 当前读自带的**行排他锁（X锁）**让并发请求排队串行执行；同时利用 `count > 0` 的乐观锁思想做最后校验，一旦库存为0，后续拿到锁的请求也无法更新成功。

## 核心薄弱点四：高级索引机制

### 1. 索引下推 (Index Condition Pushdown, ICP)
*   **目的**：减少回表次数，降低磁盘 I/O。
*   **原理**：在联合索引查询（如 `name LIKE '张%' AND age = 20`）中，原本引擎层只管过滤 `name`，拿到一堆主键后回表，再由 Server 层过滤 `age`。有了 ICP 后，引擎层在遍历索引树时，直接顺便把 `age = 20` 给过滤了，只有满足条件的才去回表。

### 2. 前缀索引 (Prefix Index)
*   **场景**：给 `VARCHAR(255)` 的长字符串（如邮箱、URL）建索引。
*   **方案**：为了防止 B+ 树节点过大，计算字段的“区分度”（通常前 10-20 个字符区分度就能达到 90% 以上），只对字符串的前 N 个字符建立索引。
*   `ALTER TABLE t ADD INDEX idx_col(col(N));`
*   **缺点**：无法用于 `ORDER BY` / `GROUP BY`，也无法作为覆盖索引（必然需要回表验证完整字符串）。
